### STORY-026b — Central auth proxy (Next.js 16) ✅ (2026-06-12)

**Shipped**
- `src/proxy.ts` — a single fail-closed auth gate. (Next.js 16 renamed `middleware` → `proxy`.)
  - `/dashboard/**` → redirect to `/login` if no Supabase session cookie present.
  - `/api/crm/**`, `/api/products/embed-all`, `/api/merchants/rotate-token` → 401 unless a
    well-formed `X-Merchant-Api-Key` (`lmd_` prefix) is present.
- Matcher **excludes** channel webhooks (`/api/webhook/**`, `/api/webhooks/**`), cron, `/api/health`,
  and `/api/merchants/register` (public onboarding).

**Decisions**
- **Optimistic checks only, zero network I/O** — per Next.js 16 guidance (proxy "should not be used
  as a full session management or authorization solution" and must not do slow data fetching). The
  proxy inspects cookies/headers in memory; the **authoritative** checks stay at the data layer
  (`getDashboardContext()` for the dashboard, `resolveMerchantFromApiKey()` for the APIs). Defense in depth.
- **Excluded `/api/products/import` and `/api/products/[id]/embed`** — they accept a *legacy* auth
  scheme (`x-merchant-id` + `x-api-key`=`MERCHANT_API_KEY`), so a blanket `X-Merchant-Api-Key` gate
  would reject valid legacy callers. They keep their per-route checks until the legacy scheme is retired.
- Webhooks deliberately off the matcher: they authenticate via their own secret-token / HMAC, and
  keeping them off the proxy means the **customer-facing bot hot path adds exactly zero proxy overhead**.

**Trade-offs accepted**
- The proxy does **not** refresh the Supabase session (no `getUser()` in the proxy) — that would add a
  ~1 auth round-trip per dashboard navigation, which the Next.js 16 docs explicitly discourage. Client
  SDK auto-refresh + the layout's authoritative `getUser()` cover validity. Residual: a merchant whose
  access token expires while away may get one extra `/login` bounce on next hard navigation. Acceptable
  for a low-frequency dashboard; revisit with header-passed identity if it becomes noticeable.

**Verification (runtime — `next dev`, live probes)**
- `/api/health` (unmatched) → `200` (proxy doesn't touch non-matched routes; app boots).
- `/api/crm/orders` no key → `401 {"error":"Unauthorized"}` (proxy fail-closed).
- `/api/crm/orders` malformed key → `401` (proxy rejects non-`lmd_`).
- `/dashboard` no cookie → `307 → /login` (default-deny).
- `tsc --noEmit` exit 0; `eslint src/proxy.ts` exit 0.

**Latency analysis (before → after)**
- The proxy adds **no network round-trip** on any path — only in-memory cookie/header inspection
  (sub-millisecond edge CPU).
- **Bot / webhook hot path:** excluded from the matcher → proxy never runs → **0 added latency**.
- **CRM API:** + sub-ms edge check on valid calls; unauthenticated calls are now rejected at the edge
  *before* the DB lookup → the reject path is **faster** than before.
- **Dashboard navigation:** + sub-ms cookie check; no new auth round-trip (optimistic only).
- Net: end-to-end API latency is effectively unchanged (well under 1 ms added); structurally no new
  I/O was introduced. (Local-dev observed; exact production edge overhead is region/plan dependent.)

  non-admin inbound message; null-guarded (`IS NULL` filter) so the write fires at most once per

  customer response; emits structured log event `funnel.first_customer_message`.
- `form-start` step maps to Vercel Analytics `/onboard` page view (already instrumented; no DB event
  needed).

**Funnel query (run in Supabase SQL editor):**
```sql
SELECT
  COUNT(*)                          AS total_registered,
  COUNT(webhook_configured_at)      AS webhook_configured,
  COUNT(first_customer_message_at)  AS sent_first_message
FROM merchants WHERE is_active = true;
```

**Decisions**
- Two nullable columns on `merchants` rather than a new events table — zero schema complexity,
  queryable with a single aggregate, sufficient for the ">70% onboarding completion" launch criterion.
- `first_customer_message_at` tracks the first *customer* message, not admin commands — this measures
  whether the merchant successfully invited a real customer, not whether they tested `/register` themselves.
- Fire-and-forget stamp (non-awaited) in the webhook handler — the funnel write is diagnostic;
  it must never add latency or error-surface to the customer-facing bot hot path.

**Trade-offs accepted**
- New columns are cast via `as any` until `supabase db push` applies migration 015 and

### STORY-032 — Referral program ✅ (2026-06-13)

**Shipped**
- Migration `017_referral_codes.sql` — adds `referral_code TEXT UNIQUE` (outbound share code)
  and `referred_by_code TEXT` (inbound: code used at registration) to `merchants`.
  Backfill generates codes for existing merchants: `LMD` + 5 chars from MD5 of their UUID.
  Partial index on `referred_by_code` for fast attribution lookups.
- `generateReferralCode()` in the register route — generates `LMD` + 5 unambiguous uppercase
  alphanumeric chars (no 0/O/1/I) for each new merchant.
- `POST /api/merchants/register` — accepts optional `referral_code` field (validated as
  `/^LMD[A-Z0-9]{5}$/`); writes `referred_by_code` on the new merchant row; if the referral
  code belongs to an active merchant (and is not self-referral), extends their `trial_ends_at`
  by 30 days from its current value (floor: now, so an expired referrer still gets value).
  Code-not-found is a soft warning — registration is never blocked by a bad code.
- `referral_code` included in the register API response so the onboard success screen can
  display it immediately.
- `/onboard` page — referral code input field (optional, auto-uppercases); `?ref=LMDXXXXX`
  query param pre-fills the field when following a shared link; success screen shows the new
  merchant's own referral link with a copy button and explains the reward.
- `/dashboard/referrals/page.tsx` — server-rendered: shows merchants-referred count, total
  days earned, trial days remaining; referral link copy button (client component).
- Referrals nav link added to `layout.tsx`.

**Decisions**
- **Reward trigger: registration** (not first payment) — Paystack subscription billing webhooks
  are not yet wired; applying the reward at registration is the simplest defensible trigger and
  matches how most B2B SaaS referral programs work at early stage. Document as "update to
  payment trigger when subscription billing is wired."
- **Reward mechanism: extend `trial_ends_at`** — maps directly to the existing billing model
  without adding a new column. `Math.max(Date.now(), existing_trial)` ensures a referrer who
  has already expired still gets full value from the reward.
- **Soft failure on invalid codes** — registration proceeds even if the referral code is not
  found. Logging a warning is sufficient; blocking a registration over an expired or mistyped
  code would be a bad UX.
- **Self-referral guard** — `.neq('id', merchant.id)` prevents a merchant from using their
  own code during a second registration with a different email.

**Trade-offs accepted**
- `as any` casts used until migration 017 applied + types regenerated (same documented pattern).
  Removed after user confirmed types regenerated; `tsc --noEmit` exits 0.
- No referral history table — `referred_by_code` on `merchants` is sufficient to reconstruct
  the full attribution graph at query time. A dedicated events table adds no pilot value.

**Verification**
- `tsc --noEmit` → exit 0 (clean after cast removal).

**Deferred / tech debt**

## Sprint 7.5 — Subscription billing + cron hardening (2026-06-13)

Post-close addendum triggered by a production report: the `cart-recovery` and `payment-expiry`
crons showing "Failed (HTTP error)" on the Vercel dashboard, plus a request to wire merchant
subscription billing through the existing Paystack webhook.

### Cron failures — diagnosis & fix ✅

**Diagnosis (from Vercel runtime logs):** both crons reject at the auth gate with **HTTP 401**.
- The running code used `if (secret !== process.env.CRON_SECRET)`. This has two faults:
  1. **Silent-open** when `CRON_SECRET` is unset (`undefined !== undefined` → false → runs
     unauthenticated). Logged as **VULN-003** (SECURITY.md).
  2. It still 401s in production because Vercel only injects
     `Authorization: Bearer <CRON_SECRET>` when `CRON_SECRET` is present in the deployment's
     env — a 401 means the bearer isn't reaching the function (env var missing in the
     Production scope, or set after the running deployment without a redeploy).

**Shipped**
- Hardened both crons to the fail-closed standard `if (!expected || secret !== expected)`
  (matches `merchant-digest` + `bot-health`). Closes VULN-003.

**Scheduler moved to cron-job.org (Hobby plan):** the `crons` block was **removed from
`vercel.json`** — Vercel Hobby caps cron jobs and only fires them once per day, too coarse for
cart-recovery/payment-expiry (which want 15-min cadence). All four crons now run on
[cron-job.org](https://cron-job.org), each sending `Authorization: Bearer <CRON_SECRET>` as a
custom header. Setup documented in `docs/CRON_SETUP.md`. The endpoints are unchanged — they're
scheduler-agnostic bearer-authenticated GETs, so this is purely a scheduling-layer move.

**Operational fix (owner action — the actual 401 cause):**
1. Set `CRON_SECRET` in Vercel → Settings → Environment Variables, scope **Production** (a long
   random string), and paste the same value into each cron-job.org job's `Authorization` header
   (as `Bearer <secret>`).
2. **Redeploy** so the running deployment reads `CRON_SECRET`.
3. Create the four cron-job.org jobs per `docs/CRON_SETUP.md` (cart-recovery + payment-expiry
   every 15 min; bot-health 06:00 UTC; merchant-digest 07:00 UTC).
4. Note: opening a cron URL in a browser will *always* 401 (no bearer) — that is correct.

### Bot-token decryption bug ✅ (functional, high-impact)

**Found while in these files:** `cart-recovery`, `payment-expiry`, and the **Paystack webhook**
all sent to Telegram using `merchant.telegram_bot_token` **raw**, but tokens are encrypted at
rest (Sprint 5). Only `merchant-digest` decrypted. Net effect: for every self-service-registered
merchant, cart-recovery nudges, expiry notices, **and order payment-confirmation messages were
silently failing to send** (the Telegram client swallows the error).

**Shipped**
- All three sites now `safeDecrypt(token) ?? token` before sending (the `?? token` fallback keeps
  any legacy unencrypted token working). Pattern matches `merchant-digest`.

**Shipped**

  `subscription_renews_at`, `referral_rewarded_at` to `merchants`; indexes on subscription/customer
